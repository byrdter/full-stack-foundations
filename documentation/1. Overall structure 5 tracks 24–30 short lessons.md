## 1. Overall structure: 5 tracks, ~24–30 short lessons

Each lesson is ~15–25 minutes of work:

- 3–5 min concept
- 10–15 min hands-on
- 2–5 min reflection / PIV connection

### Track 1 – Big Picture & PIV Mindset (4 lessons)

**L1. What is AI Coding? (Stack-agnostic)**

- Why “AI coding” ≠ “just using ChatGPT”
- Guardrails and reliability as the main theme
- Brief tour of stacks: Python / Java / Go / JS, etc.

**L2. The PIV Loop (Plan → Implement → Validate)**

- Walk through the PIV diagram you shared
- Show micro-examples (non-coding: making dinner, planning a trip)
- Connect PIV to “trust but verify” with AI

**L3. Anatomy of a Modern Backend (Language → Framework → DB → Infra)**

- Conceptual: what each layer does
- For each layer: common options (Python vs Java vs Go, etc.)
- Why this course chooses: Python + uv + FastAPI + Postgres

**L4. What Is an AI-Optimized Codebase?**

- The 6 pillars from the Excalidraw: tests, logging, infra, DB, monitoring, shared patterns
- How this is “PIV pre-planning”: preparing the stage for months of loops

> **Teacher notes (for this track)**
>
> - Keep it high-level and encouraging; no commands yet.
> - Ask them to *say back* in their own words what PIV is and why we’re choosing Python.

------

### Track 2 – Environment & Tooling (6 lessons)

**L5. Installing the Tools**

- Python, uv, VS Code, Git, GitHub account
- Generic note: alternative tools (Poetry, pip, JetBrains, etc.)

**L6. Your First Python Project with uv**

- `uv init`, running a simple `main.py`
- Explain what a package manager does, and alternatives in other languages

**L7. Git & GitHub Basics for AI Projects**

- `git init`, `commit`, `push` to GitHub
- Why version control is *critical* when an AI is generating code

**L8. First Mini PIV Loop in Code**

- Plan: “Create a script that prints a message”
- Implement: they write it (optionally with AI help)
- Validate: run it, commit with a good message

**L9. Using VS Code Effectively**

- Explorer, terminal, extensions, basic debugging
- Show how this generalizes to other IDEs

**L10. Letting the AI Help with Setup (but in a controlled way)**

- Use a *very small* standard prompt to ask the AI:
   “Help me create a uv project with a main.py that prints X”
- Compare its output to what they just did manually

> **Teacher notes (for this track)**
>
> - Emphasize: “You should be able to *fix* what the AI does, not just copy-paste.”
> - Watch for overwhelm; keep each lesson’s command set tiny.

------

### Track 3 – Guardrails: Tests, Types, and Logs (7 lessons)

**L11. Why Tests Matter (especially with AI)**

- Show a simple bug the AI might create
- Conceptual: unit tests vs manual testing; parallels in Java/JUnit, etc.

**L12. Adding pytest with uv**

- Install pytest, write one tiny test for a pure function
- Run `pytest` and see green/red

**L13. Breaking and Fixing Code via Tests**

- Deliberately break the function
- Watch tests fail, fix, rerun
- Tie directly to “Validate” in PIV

**L14. Linting & Formatting with Ruff**

- What linters are; other tools (Flake8, ESLint, Checkstyle)
- Install Ruff, run it, fix a few warnings

**L15. Types & MyPy/Pyright**

- Concept: static vs dynamic typing (contrast with Java/Go)
- Add type hints to earlier code, run MyPy or Pyright, fix issues

**L16. Logging Basics**

- What logs are, where they go, simple `logging` calls
- Show a naive log line vs a useful one

**L17. Structured Logging for AI-Optimized Codebases**

- Introduce the workshop’s structured logging pattern (domain/component/action)
- Log a small fake workflow and read the output like a story

> **Teacher notes (for this track)**
>
> - Always tie back to: “These are the *eyes and brakes* we give to our future AI helper.”
> - After L17, ask them: “If an AI added a bug, how would tests/logs/types help you catch it?”

------

### Track 4 – Architecture & Vertical Slices (5–6 lessons)

**L18. Why Architecture Matters for AI & Humans**

- Compare “one giant file” vs structured folders
- General architectures: layered, ports/adapters, vertical slice

**L19. Intro to FastAPI & First Endpoint**

- Install FastAPI/uvicorn, create a simple `GET /ping`
- Mention alternatives: Django/FastAPI/Flask, or Spring Boot, Gin, Express

**L20. Vertical Slice Architecture Overview**

- Show a diagram: `core/`, `shared/`, `features/todos` etc.
- Explain why this helps AI navigate and generate consistent code

**L21. Building Our First Slice (Read-Only)**

- Create one feature folder with route + service + schema + test
- Walk through file responsibilities slowly

**L22. Adding DB & Migrations (Conceptual + Tiny)**

- Explain relational DBs and why Postgres
- Create one simple table via SQLAlchemy + Alembic (or even stub DB first)
- Alternatives: MySQL, SQLite, MongoDB, etc.

**L23. Health Checks & Monitoring Endpoints**

- `/health`, `/health/db`, simple error handling
- Tie to observability pillar from AI-optimized diagram

> **Teacher notes (for this track)**
>
> - Keep reminding them: “The AI will be so much better inside this structure.”
> - Ask them to navigate the directories verbally: “Where do you think the AI should put X?”

------

### Track 5 – Working with an AI Coding Assistant & Capstone (6–7 lessons)

**L24. The Standard Prompt Pattern**

- Walk through the workshop’s prompt:
   context → install → implement → tests → summary/commit
- Map each part to PIV and to the tools they’ve learned

**L25. Context Engineering for Code**

- What to feed the AI: file paths, snippets, README, docs links
- Feature-based context vs whole-repo dumps

**L26. Using AI to Add a New Test (Small)**

- Plan: describe the behavior in natural language
- Implement: ask AI to write a pytest for it
- Validate: run tests, maybe adjust

**L27. Using AI to Modify a Vertical Slice**

- Add a new field or endpoint with AI’s help
- Watch for where it guesses wrong; use tests/types/logs to catch it

**L28. Capstone Planning: Design a Simple AI Feature**

- Choose something like “Summarize notes”, “Classify tasks”, or “Rewrite text”
- Do a *planning-only* PIV:
  - Inputs/outputs
  - Where it lives in the architecture
  - What tests will look like

**L29. Capstone Implementation with AI**

- Use the standard prompt to have the AI scaffold the slice
- You (and your sons) run the tools to keep it honest

**L30. Capstone Validation & Reflection**

- Run tests, manual calls, inspect logs
- Reflection questions:
  - Where did the AI help most?
  - Where did tests/types/logs save you?
  - How would this scale to bigger projects?

> **Teacher notes (for this track)**
>
> - Emphasize that *they* are the engineer; the AI is a powerful but fallible intern.
> - Invite them to propose improvements to prompts as they gain experience.

## 5. Reusable lesson template (for your Loom recordings)

For each lesson we eventually write, we can follow a consistent, easy-to-record format:

1. **Hook (1–2 min)**
   - Why this matters for AI coding / PIV.
2. **Concept (3–5 min)**
   - One or two key ideas, with analogies and mention of alternatives (Python vs others, uv vs Poetry, etc.).
3. **Guided Hands-On (10–15 min)**
   - Concrete steps: commands, file changes, expected output.
4. **Mini PIV Reflection (2–3 min)**
   - “What did we Plan, what did we Implement, how did we Validate?”
5. **Teacher Notes**
   - Common pitfalls, extra examples, optional “stretch” ideas.

We’ll flesh this out lesson by lesson once you’re happy with the structure.
